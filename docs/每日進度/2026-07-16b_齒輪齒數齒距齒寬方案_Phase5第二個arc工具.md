## 一、設計與拆解

齒輪工具 = 弧卡尺量測環（重用 arc ROI 基礎建設）＋ 純 Domain 齒輪分析器。演算法（spec §6）：弧卡尺邊點 → 依 amplitude 正負分「進齒/出齒」→ 配對成齒（含 0/2π 環繞）→ 齒數/齒距/齒寬統計 + 三條件判定（齒數==標稱、齒距最大偏差≤公差、齒寬最大偏差≤公差）。範圍定為 A+B 一起做；CSV 採三列（齒數/齒距/齒寬）。

多條件判定套用 **GD&T 先例**：不走單值 `ToleranceSpec`/`GetMeasuredValue`，改在 `MeasurementWorkflow` 開專屬分支直接發三個 `ItemJudgment`，且分支以「配方工具的 `tool.Gear != null`」為鑰、**不看 `r.Gear.Success`**——成功/失敗全歸它管，量測失敗也發一列，**永遠到不了 `GetMeasuredValue`**（避免 arc 那個「回 0 → `[0,0]` 判 OK 說謊」的坑重演）。

## 三、GUI 驗收逼出的 3 個真缺陷（單元測試 + 全鏈靜態審查都看不到）

使用者 GUI 驗收，依序抓到：

1. **試測 vs 主頁顯示不同**（同 arc 那類 parity 疑慮）：查證後判定**主頁才對**——主頁畫「齒中心」結果（20 十字＝齒數、缺齒洋紅），編輯器試測沿用弧形分支畫「原始邊點」。使用者選「保留分工（試測=調機邊點、主頁=結果）+ 加註說明」；兩處繪製點都加註「刻意分工、勿當 parity bug」。

2. **齒中心十字畫在白色齒隙、不在黑齒上**（真的 correctness bug）：根因是 **HALCON 弧掃描 handedness**——`gen_measure_arc` 於 `AngleExtent>0` 為 CCW、`measure_pos` 的 amplitude 號沿弧主軸方向定義；但分析器用 `θ=atan2(row−cr,col−cc)` 排序＝ **θ=−φ**（與掃描反向），故暗齒的「進齒邊(light→dark)」在分析器眼中是**正** amplitude，原本寫 `entering=齒為暗?amp<0` 判反、配對偏移半週期 → **中心落齒隙、齒寬量到齒隙寬（~10° 而非 ~8°）**。三項判定因對稱剛好都 PASS 遮住它，只有 overlay 讓真相現形。**先請使用者取消「齒為暗」即時確認方向**（十字跳到黑齒→證實），再套正式修正：`entering=齒為暗?amp>0:amp<0`，並把合成測試邊點正負號翻成 HALCON 真實慣例（測試對稱、全數仍成立，但現在真的守著此接縫）。查證用離線 HALCON 手冊 `measure_pos`/`gen_measure_arc` 條目。**既有 arc 工具不受影響**（只數邊點總數、不做極性配對）。

3. **缺齒洋紅十字畫在缺口旁的齒上、不在缺口中**：`MissingToothHintsDeg` 存 `centers[t]`（大間隙前一顆齒中心），改存間隙中點 `centers[t]+pitch/2`（=缺齒該在的位置）；測試強化為斷言提示點落在 90°（被移除的第 5 齒）。

## 四、教訓

- **合成單元測試綠 ≠ 整合正確**：Part A 的 amplitude 正負號是我自己「定義」的，沒對到 HALCON 實跑；全鏈終審也是靜態、看不到 runtime handedness。HALCON adapter 無單元測試 → 這類接縫只有 GUI 實跑會現形。**新的 arc-based 極性配對工具（PCD/孔陣列）務必記得這個 handedness：θ=atan2(row−cr,col−cc)=−φ_HALCON。**
- **overlay 是最好的接縫驗證**：齒中心/缺齒兩個 bug 判定都 PASS，全靠標記位置暴露；每個新方案都該把「中間量」畫出來給人看。
- 極性/正負號類修正前先造最便宜的即時情境確認（取消勾選 → 看十字跳位），再動手，避免把 handedness 改反。
