## 一、路線：從 gear 挖到「配方零弧支援」的隱藏成本

使用者想做「齒數/齒距」方案。brainstorm 定案演算法（雙極性配對、齒為暗、ROI 中心、三條件判定）後，查證發現一個會改變範圍的事實：**配方對弧形 ROI 零支援**——`RoiGeometry` 只有 rect2、`RecipeRunner` 無 arc 分支、`RecipeEditor` 只擷取 rect2、`DetectEdgesOnArc` 連 `IEdgeDetector` 介面都沒上。A3 互動弧卡尺當初只做進 MainWindow 邊緣分頁，從未進配方。

因此「gear 做成配方工具」實含兩個子系統。依 scope 原則拆開：**先做「弧形 ROI 進配方」基礎建設（本次）**，gear 方案 spec 先寫存放（下游）。基礎建設一次解鎖所有 arc-based 方案。gear spec 也記錄了它對本基礎建設的依賴。

## 二、subagent-driven 執行（6 task）

每 task 派全新 implementer + 審查。關鍵決策與亮點：

1. **schema v7**：`MeasurementTool.ArcRoi`（重用既有 `ArcMeasureRoi`，加性 nullable、向後相容）+ Validator 規則 + Domain 測試（round-trip、向後相容）。
2. `DetectEdgesOnArc` 加進 `IEdgeDetector<TImage>`（唯一既有介面改動；查明只有 2 個實作者，含會編譯失敗的測試 Fake）。
3. **`ArcRoiTransform` 放 Application 層**（非 RecipeRunner）+ 帶旋轉姿態對齊測試。**關鍵架構理由**：兩個測試專案都不參考 App.Wpf，放 Runner 內就無法被測試守護——這正是 Phase 2 驗證洞的**結構性成因**。生產與測試共用同一份程式碼，且 implementer 依要求**刻意種 bug 證明測試會 bite**。
4. RecipeRunner Pass 1.2 弧分支（跟隨姿態、邊數→既有判定）。
5. RecipeEditor 弧面板 + 互動擷取（最大塊）。
6. MainWindow overlay 畫弧 + 邊點 + 標籤。

## 三、審查攔下的真缺陷（多數源自我的計畫/spec）

兩階段審查在合併前抓到 4 個真缺陷，**多數是計畫/spec 的錯、implementer 忠實照做**：
- schema bump 6→7 打破既有硬寫 `==6` 的測試（implementer 自己發現並最小修正）。
- 計畫叫 arc 填 rect2 `Roi` → MainWindow 無條件把非 null `Roi` 畫成 (0,0) 橘框（審查抓到）。
- `DeepCopyTool` 漏複製 `ArcRoi` → 載入/存檔遺失＝**出貨阻斷**（Task 5 implementer 自己發現，spec 沒想到）。
- arc 工具用 `EdgeParameters` 但編輯器 Edge 面板對 arc 隱藏（Task 5 補上）。

## 四、全鏈終審抓到 GUI 看不見的 CSV bug

6 task 完成後做**整體終審**（每個 task 都是隔離審查、沒人檢查端到端接縫）。抓到一個**只有全鏈審查才可能發現、且 GUI 驗收看不出來**的阻斷缺陷：

`MeasurementWorkflow.GetMeasuredValue` 是個 tool-type switch，**其餘一律 `return 0`**，且 CSV 的判定是用該值**重新判定**。弧工具沒 case → CSV 以 MeasuredValue=0 重判 → **CSV 那列與畫面/OK-NG 統計互相矛盾**（統計讀 `r.IsOk` 繞過 switch，所以畫面永遠對、只有 CSV 說謊）。**GD&T 踩過同一個坑**（`MeasurementWorkflow.cs:198-200` 有註解）。修：加 `if (r.ToolType == "arc") return r.ArcEdgeRows.Count;` + arc 的 `Unit="count"`。spec §4.7 那句「零新程式碼」已更正並寫成對後續的警告。

## 五、GUI 驗收逼出的一批 UX parity 缺陷

使用者 GUI 驗收，比對 MainWindow 弧卡尺與編輯器弧工具，抓到「很low」的一批缺陷（我 under-scope 了）：①編輯器弧編輯**只有光禿禿綠把手、沒有橘/黃環帶**（環帶由 MainWindow 私有的 `DrawFittingLayers` 畫、編輯器沒複製）；②**試測鈕對 arc 禁用**（我故意延後、判斷錯又沒告訴使用者）；③角度是弧度不是度。

派獨立稽核找同類缺陷，再多找到 4 項（半徑/環寬可歸零、拖曳起始角未正規化、無載影像防呆、標籤中英混雜）。**其中一項是 HIGH**：直接放行試測會讓 `OnTrialMeasure` 讀 arc 的全 0 `Roi` 畫出 (0,0) 退化框。彙整成 7-fix 一次修完（抽共用 `DrawArcBand`、編輯器裝環帶 overlay、放行並修試測 overlay、deg 顯示、minimum、正規化、防呆、標籤），審查逐一驗過 deg 無雙重轉換、overlay 生命週期不誤清 MainWindow、DRY pixel-identical。使用者複驗全 PASS。

## 六、教訓（下游 gear 直接受用，已入 memory）

- **任何新工具型別進配方，必須同時檢查 `MeasurementWorkflow.GetMeasuredValue`**，否則 CSV 報 0（GD&T、arc 都踩過）。
- **UI 新工具要對照既有兄弟工具逐項查「半殘/降級」**：overlay 層、試測、單位、面板可見性、DeepCopy、防呆。
- **單一 task 隔離審查看不見整合接縫；GUI 驗收看不見 CSV**——大功能收尾要有「全鏈終審」這一步。
- 計畫/spec 的 snippet 會錯；implementer 該把它當參考而非聖旨，並揭露每個偏離（本次多次因此救回）。
