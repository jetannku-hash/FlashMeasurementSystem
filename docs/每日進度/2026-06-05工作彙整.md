## 2. Edge Detection bug 追蹤（多輪）

使用者最初回報：

```text
FAIL | 邊緣檢測異常 : Index out of range
```

實際追下去發現是多個獨立問題層層疊加，分多輪解決。

### 2.1 Off-by-one：`edgeDistance[i].D` 越界（使用者自修）

**Root cause**

HALCON `measure_pos` 規範（reference L3761-3763）`Distance` tuple 比 RowEdge/ColumnEdge **少 1 個元素**（描述「相鄰邊緣間的距離」，N 條邊 = N-1 個距離）。原迴圈 `for (i = 0; i < edgeRow.Length; i++)` 訪問 `edgeDistance[i].D`，最後一圈會越界。單邊（只找到 1 個 edge）的情境下 `edgeDistance.Length == 0`，第一圈就爆。

**修正**

```csharp
Distance = (i < edgeDistance.Length) ? edgeDistance[i].D : 0.0
```

**涉及檔案**

- `src/FlashMeasurementSystem.Halcon/EdgeDetection/HalconEdgeDetector.cs`

### 2.2 `DetectEdgesSubPix` 不填 EdgePoints

**問題**

亞像素路徑只 `CountObj` 拿輪廓數量、`Success = (count > 0)`，**從不解析 XLD 控制點塞 `result.EdgePoints`**。結果 UI 表格永遠空、十字 overlay 也畫不出。

**修正**

對每條 XLD 輪廓 `SelectObj` → `GetContourXld` 拿 (Row, Col) → 探查 `contrast` / `response` / `edge_amplitude` 任一可用屬性填 Amplitude → Distance 用相鄰控制點歐氏距離（最後一點 0.0）。新增 `ProbeAmplitudeAttribute` helper 在第一條輪廓上確定屬性名稱、後續沿用避免重複 query。

**涉及檔案**

- `src/FlashMeasurementSystem.Halcon/EdgeDetection/HalconEdgeDetector.cs`

### 2.3 舊版 `FlashMeasurementSystem.exe` 殘留（這不是 source bug）

**Root cause**

bin 目錄同時存在兩個可執行入口：

| 檔案 | 日期 | 大小 | 來源 |
|---|---|---|---|
| `FlashMeasurementSystem.exe` | 6/1 | 8 KB | AssemblyName 改名前的編譯殘留 |
| `FlashMeasurementSystem.App.Wpf.exe` | 6/5 | 48 KB | 目前正確產物 |

兩個都引用 halcondotnet、entry point 都是 `FlashMeasurementSystem.Program.Main`、**都能跑 detect**。使用者若點到舊 exe，前面 source 修法都不在裡面 — 看到一樣的錯誤訊息。

**追蹤證據**

使用者連續 3 次看到**完全相同**的 "Index out of range" 訊息。source 修法都已編入 dll、Reflection 確認 method 存在 — 唯一合理解釋是執行的是舊 exe。

**修正**

把 10 個舊命名檔案（`FlashMeasurementSystem.exe` / `.pdb` / `.exe.config` × bin/obj × Any CPU/x64）移到 `_obsolete_build_20260605/` 備份目錄。Rebuild 後 bin 只剩新 exe。

**教訓**

source 修法部署不等於使用者執行的程式有改 — 對改過 `AssemblyName` 的 WinForms 老專案要警覺檢查 bin 目錄殭屍 exe。

### 2.4 RGB 多通道影像被 silently 丟棄

**Root cause**

使用者影像是 `5120×3840` **3 通道 RGB byte**。HALCON 17.12 reference L3781 / L51523 把 `measure_pos` / `edges_sub_pix` 的 Image 參數明定為 `singlechannelimage`。多通道影像**不會丟 HalconException**，而是 silently 回空 tuple（log 證實 `lenRow=0`），讓使用者誤以為「detect 沒問題只是沒邊」。

**修正**

detector 入口加 `EnsureSingleChannel` helper：

- channels == 1 → 直接用原圖
- channels == 3 → `image.Rgb1ToGray()`
- 其他通道數 → `image.AccessChannel(1)` fallback

轉換後的新 HImage 在 `finally` block dispose（原圖不動）。`DetectEdges` 跟 `DetectEdgesSubPix` 兩條路徑都套用。Log 多一行 `CHANNEL-CONVERT rgb1_to_gray from ch=3`。

**涉及檔案**

- `src/FlashMeasurementSystem.Halcon/EdgeDetection/HalconEdgeDetector.cs`

### 2.5 `FromBounds` 方向邏輯（兩次翻轉、最終用 fallback）

**Root cause**

`EdgeDetectionRoi.FromBounds` 想依使用者拖出的 ROI 形狀（寬 vs 高）自動推 measure_pos 主軸方向，但**單從 ROI 形狀無法決定**：

| 使用者拖 | 可能想量 |
|---|---|
| 橫長 ROI | 水平邊（壓在邊上）**或** 左右兩條垂直邊（卡尺量寬度）|
| 縱長 ROI | 垂直邊 **或** 上下兩條水平邊（卡尺量高度）|

第一次修法把邏輯翻成「橫長→Phi=π/2 找水平邊」，解決了「橫長 ROI 跨水平邊」case，但破壞了「橫長 ROI 跨左右垂直邊」case。

**最終修正**

detector 內加 **fallback rotation**：

- PRIMARY：用 FromBounds 推的方向
- 找 0 邊時自動 rotate 90°（swap L1/L2、Phi 加 π/2）做 FALLBACK
- 取找到邊的那次結果

兩種 case 都能涵蓋。Log 區分 `PRIMARY MEASUREPOS` 跟 `FALLBACK MEASUREPOS` 方便事後 trace。

**對應實作**

- `RunMeasurePos` helper：抽出 GenMeasureRectangle2 + MeasurePos + tuple 解析
- `MeasureAttempt` struct：帶回 edges + exception
- `NormalizeToPositiveAxis` helper：把 fallback Phi 壓到 `[0, π)` 確保 edge 排序方向一致

**涉及檔案**

- `src/FlashMeasurementSystem.Domain/EdgeDetection/EdgeDetectionRoi.cs`
- `src/FlashMeasurementSystem.Halcon/EdgeDetection/HalconEdgeDetector.cs`

### 2.6 加入詳細 log 機制

之前每次 detect 失敗只能靠使用者口頭描述，缺乏可追蹤證據。加入：

- 寫到 `data/logs/edge_detection.log`（從 `AppDomain.BaseDirectory` 往上找 sln 定位；找不到退到 `%TEMP%`）
- 同步寫 `System.Diagnostics.Debug.WriteLine`
- 5MB 自動截斷避免無限長
- log 寫檔本身吞所有 exception，絕不影響 detect

每次 detect 記錄：影像尺寸 / 通道 / type、原始 ROI、clamping 結果、CHANNEL-CONVERT 步驟、PRIMARY / FALLBACK MEASUREPOS 各自 tuple 長度、END success / edge count。

這個機制在 2.4 跟 2.5 的 root cause 定位扮演關鍵 — 沒 log 看不到 `ch=3,type=byte`。

**涉及檔案**

- `src/FlashMeasurementSystem.Halcon/EdgeDetection/HalconEdgeDetector.cs`

## 3. EdgeDetection 模組 code review 與 6 項優化

review 列出 17 項可改進，按 impact × effort 分三層（A 真風險 / B UX 設計 / C 內部品質）。使用者選擇修 **A1、A2、A3、A5、B1、C1** 共 6 項。

### 3.1 A5：Fallback Phi normalize 到 [0, π)

**問題**

Fallback Phi = π/2 + π/2 = π。HALCON 對 Phi=π 跟 Phi=0 看到同一個矩形（位置大小相同），但**掃描方向相反** — 導致 edge 排序從「左→右」變「右→左」（測試發現 Edge 1 col=2058 在 Edge 2 col=1058 之前）。

**修正**

新 helper `NormalizeToPositiveAxis(rad)` 用 `rad % π` 壓到 `[0, π)`。Fallback Phi 改用這個。

### 3.2 A1：MainWindow 兜底 `catch (Exception)`

**問題**

`RunEdgeDetectionButton_Click` 只 `catch (HalconException)`。UI 端 `BindEdgeResult` / `DrawEdgeOverlay` 若丟 .NET exception（NullRef、UI thread 衝突、HObject 生命週期）會 leak 到 WinForms 主迴圈→ unhandled exception dialog。

**修正**

加 `catch (Exception ex)` fallback，訊息顯示：

```text
Edge detection failed (unexpected {ExceptionType}): {Message}
```

永遠進 SetEdgeStatus 不炸 app。

### 3.3 A2 + A3：UI 性能保護

**問題**

Subpix 一次回 793~1326 個 EdgePoint（log 實測）。`DataGridView` 無 virtualization、HALCON `DrawCross` 每點同步繪製 — UI 明顯卡。

**修正**

- `BindEdgeResult`：`MaxGridRows = 500`，超過顯示 `"showing first 500 of N"`
- `DrawEdgeOverlay`：`MaxOverlayCrosses = 200` 等間距抽樣；subpix 用 `size=3`、measure_pos 用 `size=8`

### 3.4 B1：移除 `EdgeDetectionParameters` 死欄位

**問題**

`Algorithm`、`ScanLength`、`RoiWidth`、`SubpixelMethod` 4 個欄位被 UI 賦值，但 **detector 完全沒讀** — 使用者改 UI 控制完全沒效果，UX 誤導。

**修正**

- domain 物件移除這 4 個欄位（保留 5 個真正用的：`Sigma` / `Threshold` / `Polarity` / `EdgeSelector` / `HighThreshold`）
- `MainWindow.CreateEdgeDetectionParameters` 不再 set；加註解說明 UI 控制保留為視覺 hint
- `EdgeDetectionDomainTests` 同步移除 4 個 assertion

### 3.5 C1：刪除空 event handler

`MainWindow._edgePolarityCombo_SelectedIndexChanged` 是空 method，但 Designer.cs 有綁定。Dead code。MainWindow.cs 跟 Designer.cs 都清掉。

## 6. 學習與教訓

1. **訊息字串對不上時不要假設使用者簡寫**
   第一輪推斷使用者把 `邊緣檢測異常 [12345]: ...` 簡寫成 `邊緣檢測異常 : ...`，導致連續 2 輪修錯方向。若早點請使用者貼完整訊息或檢查部署狀態，能省 2 輪。

2. **source 修法部署不等於使用者跑的程式有改**
   對改過 `AssemblyName` 的 WinForms 老專案，bin 目錄非常容易留下舊命名的殭屍 exe。今天的 `FlashMeasurementSystem.exe` (6/1) 就是這個 case。

3. **PowerShell smoke test 對 HALCON HObject 不可靠**
   `[ref] $null` 對 `out HObject` 參數 marshalling 失敗、region 沒生成；但 measure_pos 在全白圖上仍可能回某個 boundary artifact，被誤認為 fix 成功。整整繞了一輪推論建立在假象上。

4. **任何測試 setup 先驗證輸入狀態**
   應該先 `area_center` / `get_grayval` 確認 region / image 符合預期，再驗證輸出。

5. **3+ fix 失敗時要 question architecture**
   多輪修法都沒解到 root cause，要停下來問「是不是抽象本身錯了」。`FromBounds 自動推方向` 就是個錯抽象，最後用 fallback 補救而不是繼續調邏輯。

6. **加 log 是 cheapest 的 debugging investment**
   沒 log 看不到 `ch=3,type=byte`，所有後續推論都會偏。systematic-debugging Phase 1 第 4 步「加 diagnostic instrumentation」應該更早做。
