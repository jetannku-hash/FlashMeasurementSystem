## 4. 工具鏈關卡：PackageReference → vendored DLL ⭐

**計畫原訂**：Newtonsoft.Json 13.0.3，以 **PackageReference** 加在 Infrastructure。

**實作時發現的問題**（逐步查證）：
- 加 PackageReference 後 `dotnet build` 報 `CS0246: 找不到 Newtonsoft`。
- `dotnet restore` 成功：`project.assets.json` 正確解析 net48 + Newtonsoft net45 compile asset，`nuget.g.props/targets` 也生成。
- 但 **`dotnet build` 編譯期不會把套件注入成引用**——非 SDK csproj 的 PackageReference 引用注入由 VS `msbuild.exe` 提供，`dotnet build`（SDK MSBuild）沒有此機制。
- 環境查證：`msbuild.exe` 有裝（VS 2022 Pro，不在 PATH）、`nuget.exe` 無、Newtonsoft 已在全域快取。

**處置**：改用 **vendored DLL + HintPath**，與專案既有 `halcondotnet.dll` **完全相同模式**（HintPath 引用，在 `dotnet build` 下正常）。
- DLL：`lib/Newtonsoft.Json.13.0.3/Newtonsoft.Json.dll`（711 KB，自全域快取複製）
- `Infrastructure.csproj`：移除 RestoreProjectStyle/PackageReference，改 `<Reference><HintPath>`
- 優點：同時相容 `dotnet build` 與 VS msbuild、離線可重現（工廠機台不需連 nuget.org）

**取捨**（使用者裁決：維持 vendored DLL）：
- 代價：提交一個 711 KB 二進位（踩到「不提交二進位」規範，但屬第三方依賴非 build 產物，且許多 .NET Framework 工廠專案如此做）。
- `.gitignore` 加註：`lib/` 的 vendored DLL **刻意提交、勿忽略**，否則別台機器建不起來。

> 教訓：非 SDK（舊式）csproj + `dotnet build` 工作流下，第三方依賴用 **vendored DLL + HintPath** 比 PackageReference 可靠。
