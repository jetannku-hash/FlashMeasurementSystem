## 4. Distance Measurement 功能：分析與修正

### 4.1 初步分析（看 log + 讀程式碼）

使用者回報「measurement 功能使用上有問題」。先發現 **distance measurement 完全沒寫 log**（其他功能都有），所以 `edge_detection.log` 看不到任何 measurement 記錄。讀完整套程式碼後找到 6 個問題：

| # | 嚴重度 | 問題 |
|---|---|---|
| A1 | 🔴 | `measurementTypeCombo` / `contourModeCombo` 沒設 SelectedIndex；`MeasureDistanceButton_Click` 第 785 行無條件讀 `contourModeCombo.SelectedItem.ToString()`，空白時**任何 Measure 都 NullReferenceException** |
| A2 | 🔴 | measurement 跟前面工作流完全斷裂：`_latestLineFittingResult` / `_latestCircleFittingResult` 都存著卻沒用，逼使用者手動把座標抄進文字框 |
| B1 | 🟡 | `ConvertToMm` 用「方向向量加權混合 pixelSizeX/Y」，對 point-to-line（距離方向是法向量）、line-to-line、contour 的方向假設全錯，僅在非等向像素 X≠Y 時顯現 |
| B2 | 🟡 | LineToLine 用 `distance_sl`（線段對無限長直線，reference L156410），語意不對稱 |
| C1 | 🟡 | 完全沒有 log |
| C2 | 🟡 | 結果只有文字，影像上沒畫出量測元素 |

### 4.2 六項修正（使用者選擇全部一起修）

- **A1**：`OnLoad` 對 measurement 兩個 combo 加 `EnsureComboDefault`
- **A2**：加 `+ Line` / `+ Circle` 帶入按鈕，把最近成功的 fit 結果座標 append 進座標框（保留手動輸入）
- **B1**：`ConvertToMm` 改「物理座標法」——row×PixelSizeUmY、col×PixelSizeUmX 縮放後讓 HALCON 在物理空間算距離，方向自動正確；contour 用 `affine_trans_contour_xld` + `hom_mat2d_scale` 縮放（reference L161477-484 的 row→x / col→y 慣例）
- **B2**：改用 `distance_ss`（兩線段對稱距離，reference L156542）
- **C1**：仿 edge detection 加 log → `data/logs/distance_measurement.log`
- **C2**：量測成功後在影像上畫量測元素

涉及檔案：`HalconDistanceMeasurer.cs`、`MainWindow.cs`、`MainWindow.Designer.cs`、`EdgeDetectionDomainTests.cs`

### 4.3 PointToLine 數值疑問（不是 bug，是用錯 type）

使用者餵 LineToLine 的四點資料、卻用 PointToLine 量，得到 658.44px。手算驗證該值**數學正確**：PointToLine 把第 1 行當點、第 2-3 行當線、第 4 行忽略，算的是「點到一條斜跨線」的垂直距離，並非使用者本意。結論：該資料應用 LineToLine（答案 ≈1000px）。此疑問促成後續的「格式提示」功能。

### 4.4 LineToLine 代表值修正（實測發現的真 bug）

實測 adapter 各方法（已知答案座標）：PointToPoint / PointToLine / CircleToCircle / 非等向換算**全部正確**；但 **LineToLine 的代表值用 `(min+max)/2`** → 平行邊間距被灌水（300px 間距顯示 362px，因 max 是端點斜距）。修正：`DistancePx`/`DistanceMm` 改用 **min**（兩線段最近距離 = 平行邊垂直間距）。

### 4.5 overlay 優化

原本 overlay 只畫元素、看不出量的是哪段距離。重新設計用顏色分層：
- 🟢 綠 = 參考線（L / L1 / L2）
- 🟣 洋紅 = 點 / 圓心（含 label P / P1 / C1…）
- 🔴 紅線 = 真正被量測的那段距離（PointToLine 畫 P→垂足、LineToLine 畫 line1 中點→line2 垂足）
- 🟡 黃字 = 數值標在距離線中點

### 4.6 格式提示

選不同 type 時，結果區（`measureResultLabel`）自動顯示該 type 的座標輸入格式（幾行、每行代表什麼），避免餵錯 type/格式。

### 4.7 座標框加大

`measurementCoordInput` 高度 26px → 130px，並改 **Consolas 等寬字體**，多行座標好讀好對齊。

### 4.8 `+ Contour` 帶入功能

ContourMaxMin 原本只能手動逐點打字（subpix 上千點不可能手打）。加第三顆 `+ Contour` 按鈕（按鈕列改 3 欄 sub-panel）：
- 按一下把最近一次 Edge Detection 的所有 EdgePoints 一次帶入（StringBuilder 一次寫入，上千點不卡）
- 第二次按自動加空行分隔成 contour 2
- 帶入後自動切到 ContourMaxMin

### 4.9 ContourMaxMin 分隔失效 bug

使用者回報「第二次帶入沒出現空行」。根因在**解析端**：`MeasureContourMaxMin` 用 `String.Split(..., RemoveEmptyEntries)`，**會把空行整個移除**，導致「偵測空行切換 contour 2」的邏輯永遠不觸發、所有點塞進 contour 1。修正：該行改用 `StringSplitOptions.None`（保留空行），其他 Measure 方法維持 `RemoveEmptyEntries`。

### 4.10 ContourMaxMin overlay 重做

原本只畫兩條 cyan 輪廓 + 一個被蓋住的數值，看不出量的是哪段。重做：
- 🔵 cyan 點 = 兩條 contour（抽樣畫避免上千點卡頓）
- 🔴 紅線 + 兩端 cross = **最近距離 (Min)**
- 🟠 橘線 = 最遠距離 (Max)（HALCON 支援 'orange'，reference L43744 確認）
- 數值標在各距離線中點（兩輪廓之間空白處，不被蓋）

技術：用空行分成兩條 contour，抽樣暴力找最近/最遠點對，畫對應線。

## 7. 學習與教訓

1. **計算正確 ≠ 結果有意義**：PointToLine 的 658.44 數學正確但答非所問（用錯 type）；overlay 把元素清楚畫出來，使用者才能發現自己餵錯。
2. **看 log 前先確認該功能有沒有 log**：distance measurement 沒 log，是「看不到問題」的第一道牆。
3. **解析端的 `RemoveEmptyEntries` 與「空行分隔」設計直接衝突**：UI 端產生了空行，解析端卻把它吃掉，功能就壞——資料流兩端要一起看。
4. **非等向像素換算別用方向加權**：正解是「先轉物理座標再算距離」，方向自動正確，且對所有 type 統一適用。
5. **HALCON operator 參數順序 / 語意必逐字對照 reference**：`distance_sl`（線段對無限線）vs `distance_ss`（兩線段）語意不同；fit_line vs fit_circle 參數順序不同。
6. **overlay 要畫「被量測的那段」而非只畫元素**：紅線標距離 + 數值放空白處，使用者才看得懂。
