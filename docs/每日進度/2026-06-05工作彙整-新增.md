## 2. 使用者實機驗證：Line Fitting 結果解讀

### 2.1 使用者測試的場景

旋轉黑方塊影像（5120×3840 RGB），使用者拖一個橫長 ROI 跨越方塊**左右兩條垂直邊**：

- Edge Detection（measure_pos）找到 2 個 edge：
  - Edge 1: Row=2104.91, Col=2078.97, Amp=-189.0
  - Edge 2: Row=2153.79, Col=1106.69, Amp=+187.6
  - Dist=979.05（兩邊中心距離 ≈ 方塊寬度）
- Line Fitting 結果：「P1=(2104.91, 2078.97) P2=(2153.79, 1106.69) Angle=177.12° Length=972.51px RMS=0.0000px Points=2」
- 影像上顯示一條綠色斜線**連接兩 edge 中心**（橫跨整個方塊）

使用者提了 4 個問題。

### 2.2 問答摘要

**Q1：結果正確嗎？**

演算法上完全正確。Edge Detection（measure_pos 模式）對每條跨越的物理邊只回**一個中心點**，所以兩條邊得到 2 個點。fit_line_contour_xld 收到 2 個點 → 數學上剛好定義 1 條 line → 把這 2 個點連起來。`RMS=0.0000` 就是 line 完美穿過 2 個點的證明（2 點定義 1 線，無誤差）。

**但結果不符使用者的物理意圖** —— 你看到的是「左右邊中心的連線」，不是「方塊的邊」。

**Q2：操作有誤嗎？**

不算「錯」，是 **use case 跟工具能力不匹配**：

- 使用者期望「ROI 框住兩條邊 → 分別擬合成兩條 line」
- 但 measure_pos 對每條邊**只回一個中心點**（不是邊上多個 sample）
- 所以「框 2 條邊」拿到的是「邊 1 中心 1 點 + 邊 2 中心 1 點」，fit_line 無從區分

**Q3：正確操作方式**

兩條路：

**方案 A：一個 ROI 對應一條 line（最直觀）**
1. 拖一個**只跨越「一條」物理邊**的 ROI
2. Detect → Fit Line → 拿到那條邊的擬合線
3. Clear → 重畫 ROI 跨另一條邊 → Detect → Fit Line → 拿到第二條
- 缺點：要做兩次

**方案 B：切到 EdgesSubPix radio（精度更高）**
- canny 對一條邊會回幾百個亞像素控制點（之前 log 看到 793 / 1326）
- Fit Line 對密集點 fit，精度遠高於「2 個中心點」
- 這是 line fitting 的**標準工作流**：canny 提取輪廓 → 擬合幾何形狀
- 仍需「一個 ROI 對一條邊」原則才能拿到乾淨的 line

**Q4：一個 ROI 能否分別 fit 多條 line？**

**目前不行**，兩個限制：

1. `HalconLineFitter.FitLine(IList<EdgePoint> ...)` 接收**一個點集合**輸出**一條 line** —— spec out-of-scope 條目第 1 條 `Multi-line fitting` 明列不做
2. HALCON `fit_line_contour_xld` 本身也是「1 個 contour → 1 條 line」，要多條需先用 `segment_contours_xld` 切割成多段

若要做「一個 ROI 內自動切割成多條邊、各自擬合」，需要：
- 加 segmentation 步驟（依 contour 折角自動切）
- 或加群聚演算法（DBSCAN / connected component）
- domain `LineFittingResult` 要改成 `IList<LineFittingResult>`
- UI 表格要支援多行結果

屬於另一份 spec 的範圍。

## 3. ROI 互動方式討論

使用者後續問了兩個關於 ROI 的問題。

### 3.1 Q1：ROI 框能否改成可調整角度？

**技術上完全可以**，現有基礎結構已大致支援：

| 元件 | 現狀 |
|---|---|
| `EdgeDetectionRoi.AngleRad` | 已有欄位 |
| `OverlayAnnotator.DrawRectangle2(row, col, phi, len1, len2, color)` | 已能畫旋轉矩形（HALCON DispRectangle2 內建支援 phi）|
| Detector 對任意 Phi 的支援 | 已支援（fallback 邏輯用 NormalizeToPositiveAxis 處理任意角度）|
| `HWindowControlHelper` 的 ROI 互動 | **只記錄 axis-aligned 兩個對角點**，沒有角度概念 |
| `EdgeDetectionRoi.FromBounds` | 只從 axis-aligned bounding box 推 Phi=0 或 π/2 |
| UI 角度輸入控制 | 沒有 |

改造方向有三種：

**方案 A：UI 加 Angle (deg) NumericUpDown**
- 使用者拖一個 axis-aligned ROI，再用 numeric 調角度
- 改 ROI internal model + 畫圖 + add angle binding
- 最簡單、現有滑鼠互動完全不動
- 角度精確（可以設到 35.5°）但旋轉感不直覺

**方案 B：拖完矩形後出現「旋轉手把」**
- ROI 拖完顯示矩形邊上一個圓點
- 拖那個圓點 → 即時旋轉預覽（類似 Photoshop / Visio）
- 需要在 `HWindowControlHelper` 改寫滑鼠處理（pointer hit-test 區分角點 / 手把 / 邊）
- 中等改動

**方案 C：標準 HALCON ROI 編輯器互動風格**
- 第一拖：定中心
- 第二拖：定長軸方向 + 長度
- 第三拖：定短軸寬度
- 最接近 HDevelop 專業習慣
- 改動最大

**推薦方案 A**：最少改動、最快上線、角度精確。未來 B/C 可迭代。

### 3.2 Q2：為什麼不用 HALCON 內建 ROI 元件？

**短答**：HALCON 17.12 沒有「即插即用、modal-less 的 ROI editor 元件」可以直接塞進 WinForms 用。`HWindowControl` 只是「影像顯示視窗」，不附 ROI 編輯工具。

HALCON 提供的 ROI 相關 API 都有限制：

| HALCON API | 是什麼 | 為什麼不直接用 |
|---|---|---|
| `HOperatorSet.DrawRectangle2(window, out row, out col, out phi, out len1, out len2)` | 互動畫旋轉矩形 | **modal 阻塞**：呼叫後 thread 卡住直到使用者按 enter；WinForms event loop 同時不能處理其他互動 |
| `draw_rectangle1` / `draw_circle` / `draw_polygon` | 各種形狀互動繪製 | 同樣 modal blocking |
| `drag_region1` / `drag_region2` | 拖曳已存在的 region | 也是 modal、且需要先有 region 物件 |
| `HSmartWindowControl`（HALCON 較新 control） | 比 `HWindowControl` 多 zoom / pan 內建支援、部分版本有 ROI 工具 | 17.12 提供的版本功能有限；專案目前用 `HWindowControl`，換 control 是大改動會影響整個顯示層 |

工程上的決策（見 `HWindowControlHelper.cs:39` 註解）：

> // 使用標準 WinForms 滑鼠事件（Halcon 17.12 HMouseEventArgs 相容性不穩定）

意思是先前測試 HALCON 17.12 的 mouse event API 時遇到不穩定（event 觸發時機、座標轉換、threading 議題），切回 WinForms 標準 `MouseDown` / `MouseMove` / `MouseUp`。

**好處**：
- 可預期、跨 HALCON 版本相容
- 完全 modal-less，使用者可以隨時操作任何控制項
- 自己控制座標轉換，方便處理 zoom / pan
- 不受 HALCON event loop 干擾

**代價**：所有滑鼠互動（zoom / pan / ROI 拖曳 / 旋轉）都要自己寫，集中在 `HWindowControlHelper`。

**對應到 Q1**：要加角度，最務實做法是**繼續用 WinForms 標準事件、自己寫旋轉互動**，不是改用 HALCON 內建 API。理由：
- 現有 zoom（mouse wheel）/ pan（right drag）/ ROI 拖曳（left drag）的自寫邏輯穩定運作中
- HALCON 的 modal API 跟 WinForms event flow 衝突
- `HSmartWindowControl` 切換成本大且不保證解決問題

## 5. 學習與教訓

1. **跨文件 review 要對齊現況 codebase**
   兩份文件是先前寫的，假設了一些 API 名稱（`_hWindowHelper.AddOverlayAction`）跟結構 — 但今天上午剛改過 `HWindowControlHelper`，文件已過期。review 必須以**目前真實 codebase 為準**，不能假設文件寫的就是現況。

2. **參數順序 bug 是最隱性的 critical**
   `FitLineContourXld(..., ClippingFactor, Iterations)` vs `(..., Iterations, ClippingFactor)`：兩個都是數值、HALCON 不會丟例外、編譯通過、可以執行 — 但結果完全不對。這類 bug 必須靠**對照 reference 行號逐字 verify**，不能憑記憶。

3. **`SetPersistentOverlayAction` 的「替換」語意要顯式說明**
   今天 EdgeDetection A3 修法後，UI overlay 是「one persistent action 一次重畫所有」的模型。Line Fitting plan 沒意識到「多 feature 共用同一個 overlay slot」的設計，差點覆蓋掉 ROI 跟 edges。Fix line 加 review 才抓到。後續 feature（角度 ROI、circle fitting 等）都要小心同樣陷阱。

4. **演算法正確 ≠ 結果有意義**
   Line Fitting 跑出來的綠色斜線「演算法上對」，但使用者期望的是「方塊的兩條邊」。spec 內顯式列出 Known Limitations 比依賴 UI 細節暗示重要 — 文件化讓使用者預期跟工具能力同步。

5. **HALCON modal API 跟 WinForms 不友善**
   `HOperatorSet.DrawRectangle2` 是 modal blocking，這個本質限制讓「用 HALCON 內建 ROI」這條路在 WinForms 內變得困難。先前開發者選 WinForms 標準事件是正確決策，這個取捨值得文件化（已寫進 `HWindowControlHelper.cs:39` 但未來文件可以更顯眼）。
