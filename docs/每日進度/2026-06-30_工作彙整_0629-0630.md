## 二、6/29 — N1 配方驗證（Pre-run diagnostic）

### 背景
GUI 優化 backlog 12 項中實質價值最高者：**此前配方完全無驗證**，使用者能存下壞配方而不自知、靠 Run 失敗反推。對應 KEYENCE「programming 階段診斷每個量測點」哲學。計畫書：`docs/superpowers/plans/GUI建議優化項目計畫書.md` §N1。

### 實作內容
- **新增 Domain 純邏輯**（無 HALCON/UI/檔案，可單元測試）：
  - `Domain/Roi/RecipeIssue.cs` — `RecipeIssueSeverity{Error,Warning}` + `RecipeIssue`（嚴重度/ToolId/ToolName/Message）。
  - `Domain/Roi/RecipeValidator.cs` — `Validate(recipe, imageW, imageH)` 回傳 `List<RecipeIssue>`。檢查規則**精準對齊 RecipeRunner 執行語意**：零工具、ROI null/尺寸≤0、ROI 中心超界(Error)/部分超界(Warning)、反向公差(Upper<Lower)、各型別參考數量+型別（distance/angle/構造/GD&T）、GD&T 缺規格或帶寬≤0、重複 Id、自我參考、未知型別(Warning)、參考姿態全 0(Warning)。無影像時略過邊界檢查。
- **UI 閘門**：`MainWindow.EnsureRecipeValid()`，於 `RunRecipeButton_Click` 與 `OneClickMeasureButton_Click` 量測前呼叫 — **Error 阻擋並列出、只有 Warning 詢問是否繼續、無問題照常量測**。
- **測試**：`tests/.../RecipeValidatorTests.cs`，26 案例（spec 列的缺 ROI/超界/壞 RefId/反向公差/零工具等 + 後續誤報回歸）。
- csproj：old-style，手動加 `<Compile Include>`（Domain + Tests 各 1）。

### 過程中抓到的 bug（使用者回報）
- **ROI 邊界誤報**：初版用圓形上界 `reach=sqrt(L1²+L2²)`，把短邊方向也當對角線長 → 橫長 ROI 完整在影像內但靠邊緣時誤報「部分超界」，使用者把 ROI 拉回界內仍跳警告。
- **修法**：改用旋轉矩形**軸對齊外接框**（`halfCol=|L1·cosφ|+|L2·sinφ|`、`halfRow=|L1·sinφ|+|L2·cosφ|`），φ=0 時 halfCol=L1、halfRow=L2，與實際框一致。加回歸測試釘住。
- **教訓**：這正是「系統化測試 + 使用者目視驗收」互補的價值——單元測試全綠仍可能漏掉直覺性的幾何誤判。

### 驗證
- `dotnet build`（Any CPU + x64）皆 0 警告 / 0 錯誤。
- `Tests.exe` 全綠，含 `RecipeValidatorTests passed`，exit 0。
- GUI 逐項目視驗收（A 正常配方照常量測 / B 零工具 Error 阻擋 / C 反向公差 Error / D ROI 部分超界 Warning 詢問 + 拉回界內不警告），兩個入口（Run Recipe、一鍵量測）都驗。
