## 0. 跨項目共通決策（請先確認）

這幾點影響所有項目，請優先裁決：

### 0.1 C# 語言版本限制 ⚠️（已查證）
各專案 `.csproj` 設定 `LangVersion = 7.3`。**手冊範例用的 C# 8/9 語法無法編譯**，計畫一律改回 7.3：
- `Guid.NewGuid().ToString("N")[..8]`（range，C# 8）→ `Guid.NewGuid().ToString("N").Substring(0, 8)`
- `public List<X> Items { get; set; } = new();`（target-typed new，C# 9）→ `= new List<X>();`
- 不使用 record、不使用 `init` 存取子。

### 0.2 純邏輯服務放哪裡 ✅（建議）
4.8 / 4.10(簡易) / 4.12 不碰 HALCON。`Infrastructure` 專案已參考 `Application` + `Domain` 且目前幾乎是空的 → **純邏輯實作放 `Infrastructure`**，介面仍放 `Application`，DTO 放 `Domain`。保持嚴格分層（不把無 HALCON 的邏輯塞進 Halcon 專案）。

### 0.3 持久化方式 ✅（建議：零新依賴）
專案**未引用 Newtonsoft.Json**，.NET 4.8 也無內建 System.Text.Json。手冊範例的 `JsonConvert` 會引入新依賴，違反全域規範「不加新 dependency」。
**建議改用 .NET 4.8 內建 `System.Runtime.Serialization.Json.DataContractJsonSerializer`**（需在 csproj 加 `Reference Include="System.Runtime.Serialization"`，屬框架內建，非 NuGet）。
> 若你偏好 Newtonsoft 的易用性，也可，但需明確核准新增 NuGet 套件。

### 0.4 old-style csproj
所有新檔都必須在對應 `.csproj` 手動加 `<Compile Include="..." />`（專案不自動 glob）。每個項目下方都列出需新增的條目。

**性質**：純邏輯，無 HALCON。五件套對應：DTO(Domain) + 介面(Application) + 實作(Infrastructure，非 Halcon) + test + wiring。

### 檔案清單

| 2 | `Domain/Tolerance/ItemJudgment.cs` | 單項判定結果：實測、偏差、`IsOk`、`MarginPercent`、`Message` |
| 3 | `Domain/Tolerance/OverallJudgment.cs` | 整體：`AllOk`、`OkCount`、`NgCount`、`List<ItemJudgment>` |
| 4 | `Application/Tolerance/IToleranceJudger.cs` | `OverallJudgment Judge(IList<ToleranceItemInput> items)` |
| 5 | `Infrastructure/Tolerance/ToleranceJudger.cs` | 比較邏輯實作 |
| 6 | `tests/.../ToleranceDomainTests.cs` | console 測試，`Run()` 串進 `Main()` |

### 介面簽名（草案）
```csharp
// Application/Tolerance/IToleranceJudger.cs
public interface IToleranceJudger
{
    OverallJudgment Judge(IList<ToleranceItemInput> items);
}
// ToleranceItemInput：ToolId / ToolName / MeasuredValue / ToleranceSpec
```

### 關鍵邏輯（與手冊差異）
- `lower = Nominal + LowerTolerance`、`upper = Nominal + UpperTolerance`、`IsOk = lower <= v <= upper`。
- **改良手冊的邊界警告**：手冊 `margin = |deviation|/(upper-lower)` 在 nominal 偏一側時不準。改為「到最近邊界的餘量百分比」：
  `marginPercent = min(v - lower, upper - v) / ((upper - lower)/2) * 100`，越接近 0 越危險，< 20% 給 ⚠️。
- 除零防護：`tolRange < 1e-10` 時不算百分比。

### 測試重點
nominal±tol 邊界值、剛好命中上下限、超上限、超下限、零公差、接近邊界警告觸發。

### csproj 變更
- `Domain.csproj` +3 Compile、`Application.csproj` +1、`Infrastructure.csproj` +1、`tests .csproj` +1。

**工成**：低。**風險**：極低（純算術）。

## P1 — 4.11 座標系定位

**性質**：用 HALCON。真正的 feature adapter（實作放 Halcon 專案）。

### 檔案清單
| # | 檔案 | 職責 |
|---|------|------|
| 1 | `Domain/CoordinateSystem/RigidTransform.cs` | 參考→當前的剛體轉換描述（內含 `HomMat2D` 數值或 row/col/angle） |
| 2 | `Domain/CoordinateSystem/TransformedRoi.cs` | 轉換後的 `(Row, Col, AngleRad)` |
| 3 | `Application/CoordinateSystem/ICoordinateMapper.cs` | 建立轉換 + 套用到 ROI/點 |
| 4 | `Halcon/CoordinateSystem/HalconCoordinateMapper.cs` | 用 `vector_angle_to_rigid` + `affine_trans_pixel` |
| 5 | `tests/.../CoordinateSystemDomainTests.cs` | 測 DTO 預設 + 介面契約（HALCON 部分手動驗證） |

### 介面簽名（草案）
```csharp
public interface ICoordinateMapper
{
    // 由模板「參考姿態」與「當前匹配姿態」建立轉換
    RigidTransform CreateFromMatch(
        double refRow, double refCol, double refAngleRad,
        double curRow, double curCol, double curAngleRad);

    // 把參考座標系下的 ROI 中心/角度，轉到當前影像座標系
    TransformedRoi TransformRoi(
        double refRoiRow, double refRoiCol, double refRoiAngleRad,
        RigidTransform t);
}
```

### 關鍵做法（與手冊差異）⚠️
- **不照手冊手刻三角函數旋轉**（手冊假設「參考原點在 (0,0)」對一般情況錯誤）。
- 用 HALCON：
  - `HOperatorSet.VectorAngleToRigid(refRow, refCol, refAngleRad, curRow, curCol, curAngleRad, out HomMat2D)` 取得轉換矩陣。
  - 點/ROI 中心：`HOperatorSet.AffineTransPixel(HomMat2D, refRow, refCol, out row, out col)`。
  - ROI 角度：`curAngleRad - refAngleRad`（或從 HomMat2D 分解）。
- 與 `HalconTemplateMatcher.GetMatchContour` 慣例一致（已用 VectorAngleToRigid）。

### 測試重點
零旋轉純平移、純旋轉、平移+旋轉組合；用已知座標對照 HALCON 輸出。

### csproj 變更
- `Domain.csproj` +2、`Application.csproj` +1、`Halcon.csproj` +1、`tests .csproj` +1。

**工成**：低中。**風險**：低（用官方算子，避免手刻誤差）。

**性質**：簡易版幾乎純數學（兩點距離換算）。不需 HALCON、不需標定板（僅需已知尺寸標準件影像上的兩點）。

| 2 | `Application/Calibration/ICalibrator.cs` | 兩點校正 + 多點校正 |
| 3 | `Infrastructure/Calibration/PixelSizeCalibrator.cs` | 換算邏輯 |
| 4 | `Infrastructure/Calibration/CalibrationStore.cs` | 存/載 `CalibrationProfile`（`DataContractJsonSerializer`） |
| 5 | `tests/.../CalibrationDomainTests.cs` | 測換算數值與序列化往返 |

### 介面簽名（草案）
```csharp
public interface ICalibrator
{
    // 影像寬高用 int 傳入，使本介面與 HALCON/HImage 解耦、可單元測試
    CalibrationProfile CalibrateTwoPoint(
        double knownDistanceMm,
        double p1Row, double p1Col, double p2Row, double p2Col,
        int imageWidth, int imageHeight);
}
```

### 關鍵做法（與手冊差異）⚠️
- 基本換算：`pixelSizeUm = knownDistanceMm * 1000 / distPx`。
- **不照手冊的「斜向 `pixelSize/cosAngle`」**（語意可疑、易誤導）。簡易版預設**等向**（X=Y=pixelSize）。若要 X/Y 各別校正，提供「水平標準件」「垂直標準件」兩條正交量測各算一次，而非用單一斜線投影硬拆。
- FOV：`width * pixelSizeUmX / 1000`（mm）。
- 輸出的 `CalibrationProfile` 用來**取代 `DistanceMeasurementParameters` 寫死的 10.0**：由 UI 載入 profile 後填入量測參數。

### 與現有程式的銜接
- 目前 `DistanceMeasurementParameters.PixelSizeUmX/Y = 10.0`。校正完成後由 App 層把 profile 的值寫進去，不改 Domain 預設（保持向後相容）。

### csproj 變更
- `Domain.csproj` +1、`Application.csproj` +1、`Infrastructure.csproj` +2、`Infrastructure.csproj` 加 `Reference System.Runtime.Serialization`、`tests .csproj` +1。

**工成**：低。**風險**：低。**注意**：簡易版不含畸變補償，視野邊緣仍有殘差（完整版為後續里程碑）。
