## ❌ BUG #1（已撤回，非問題）— `MeasureCircleToCircle` 用 `DistancePp`

**檔案**: `FlashMeasurementSystem.Halcon\DistanceMeasurement\HalconDistanceMeasurer.cs` 第 135–164 行

**初版主張**: `DistancePp` 量的是圓心距，當兩圓半徑不同時 ≠ 輪廓最近距，疑為 bug。

**複查裁決**: ❌ **撤回，這不是 bug。** 方法簽名輸入即為 `(circle1Row, circle1Col, circle2Row, circle2Col)`——明確是圓心座標；`DistancePp` 量圓心距與方法語意完全一致。市售閃測儀「圓對圓」本就常見圓心距 / 輪廓最近距兩種模式，沒有哪個才「對」。初版是我把「看到輪廓 operator 就聯想」投射上去，缺乏根據。**不需任何修正。**

## ⬇️ BUG #2（降級為「最多加註解」）— `FitCircle` 的 `UsedPoints` 語意

**檔案**: `FlashMeasurementSystem.Halcon\CircleFitting\HalconCircleFitter.cs` 第 79 行

```csharp
result.UsedPoints = n;  // n = 全部輸入點數
```

**初版主張**: `atukey` / `geotukey` 會自動排除 outlier，`UsedPoints = n` 誤導為「全部 input 都被用於 fit」，建議改名 `InputPoints`。

**複查裁決**: ⬇️ **降級，不建議改名。** 查證發現：
- `fit_circle_contour_xld` 簽名輸出僅 Row/Column/Radius/StartPhi/EndPhi/PointOrder，**確實無 inlier 數量輸出**（此事實成立）。
- 但 `UsedPoints` 被**跨層大量引用**：`CircleFittingResult` + `LineFittingResult` 兩個 Domain 型別、兩個 adapter、`CircleFittingDomainTests.cs:41` 測試斷言、`MainWindow.cs:822,851` UI 顯示、開發手冊與 specs 多處。改名屬破壞性改動。
- spec 明寫「Quality assessment is left to the user via the RMS/UsedPoints fields」——此欄位語意本就是「輸入給 fit 的點數」，並非筆誤。

**建議**：不改名、不動 API、不碰測試。**最多**在 `CircleFittingResult.cs` 加一句 XML comment 註明「atukey/geotukey 內部排除的 outlier 不反映在此值」。**亦可完全不做**——屬文件層級吹毛求疵，非 bug。

## ⚠️ 潛在風險（維持，待確認）— `GetMatchContour` 的 Contour 生命週期

**檔案**: `FlashMeasurementSystem.Halcon\TemplateMatching\HalconTemplateMatcher.cs` 第 103–124 行

```csharp
HOperatorSet.GetShapeModelContours(out modelContours, _modelID, 1);
HOperatorSet.AffineTransContourXld(modelContours, out HObject transformed, homMat2D);
return transformed;
```

**說明**: `GetShapeModelContours` 回傳輪廓的複製，`finally` 裡 Dispose 複製體安全、`transformed` 為獨立輸出。理論上無問題，但若 HALCON 17.12 實作為共享 reference，連續呼叫 + Dispose 可能有風險。

**複查裁決**: 維持為**低優先觀察項**，目前無實證問題。若日後出現 contour 顯示異常再查。

## 🚀 優化建議（複查後逐項裁決）

### 1. `DetectEdgesSubPix` 的 `Sigma` 預設值 — 🔸 可做可不做
預設 `Sigma = 1.2`，高雜訊閃測影像可能偏敏感。**裁決**：純加 XML comment 說明調參建議即可，零風險但價值低。不改預設值（改了可能影響現有使用者）。

### 2. `MeasureLineToLine` 的 `DistanceMaxMm` — ❌ 撤回（初版事實錯誤）
**初版主張**「DistanceMaxMm 未被使用、應一併暴露」**是錯的**。查證：
- `DistanceMeasurementResult.cs:11` 欄位已存在
- `HalconDistanceMeasurer.cs:113,200` 已賦值
- `MainWindow.cs:1189-1191, 1313` UI 已顯示 Min 與 Max

初版只看了 adapter 內 `DistanceMm = DistanceMinMm` 一行就妄下結論，沒往 UI 查。**完全撤回，無需任何改動。**

### 3. Edge 點缺少輪廓歸屬資訊（`ContourIndex`）— ❌ 不做
`EdgesSubPix` 多輪廓時所有點扁平化於同一清單。**裁決**：加一個目前**無任何 caller 消費**的欄位，正好踩中全域 CLAUDE.md「避免 future-proofing / 未要求功能」。除非實際遇到「邊緣斷裂混合擬合」痛點，否則不加。**先不做。**

### 4. Log 檔截斷方式 — ✅ 已實作（本次）
**裁決**：採納。`MaxLogBytes = 5MB` 截斷時原為 `File.Delete(path)`，已改為 Move 到 `.bak`（先刪舊 .bak 再 Move），保留最近一輪歷史。
- 已修改：`HalconEdgeDetector.cs`（`edge_detection.log` → `.bak`）
- 已修改：`HalconDistanceMeasurer.cs`（`distance_measurement.log` → `.bak`）
- 副作用：磁碟用量上限由 5MB 變 10MB（本檔 + .bak），可接受。

### 5. ROI 角度正規化 — 🔸 觀察項
`NormalizeToPositiveAxis` 正規化到 `[0, π)`，`AngleLl` 結果為 `(-π, π]`。目前僅用於 edge 偵測 fallback，無實證問題。維持觀察。

### 6. `AngleMeasurement` 的 `ReferenceLength` magic number — ❌ 不做
`ReferenceLength = 100.0`。**裁決**：`angle_ll` 只取向量**方向**，長度只要非零，算出的角度**完全相同**——100 或 1000 結果一致。改成隨影像比例縮放是做白工，甚至會誤導後人以為長度影響結果。**不改**，頂多加一行 comment 註明「長度任意，非零即可」。
